# Звіт до лабораторної роботи

## Тема
Розробка backend-модуля AI-адміністратора для системи підтримки інтернет-магазину `shop-3d` та підготовка frontend-контракту для адмін-панелі.

## Мета роботи
Реалізувати серверну частину AI-адміна, який:
- працює всередині адмін-панелі;
- читає дані з бази без прямого доступу моделі до MongoDB;
- генерує відповіді для чату;
- за потреби відправляє повідомлення в реальний чат користувача або гостя;
- зберігає повідомлення в БД і надсилає їх через socket у frontend.

## Постановка задачі
У початковій версії проєкту повідомлення адміністратора могли не відображатися у чаті після reload або не потрапляли в історію через різні namespace та прив'язку до конкретного `adminId`. Також у системі не було повноцінного backend-шару для AI-адміна.

Було поставлено такі задачі:
- створити окремі API routes для AI-адміна;
- інтегрувати OpenAI через Responses API;
- реалізувати tool calling для безпечного читання даних з БД;
- забезпечити запис AI-повідомлень у MongoDB;
- забезпечити socket-доставку AI-повідомлень у віджет користувача та в адмін-панель;
- підготувати готовий frontend API-клієнт.

## Реалізовані зміни

### 1. AI admin backend
Було створено окремий сервіс `aiAdminService`, який:
- перевіряє наявність `OPENAI_API_KEY`;
- використовує `OPENAI_MODEL` або модель за замовчуванням `gpt-5-mini`;
- автоматично знаходить або створює спеціального користувача з роллю `admin` та прапорцем `isAiAssistant`;
- формує контекст діалогу;
- викликає OpenAI через Responses API;
- виконує дозволені tools;
- у режимі `reply` відправляє повідомлення в чат.

### 2. Безпечна взаємодія з БД
Модель не отримує прямий доступ до MongoDB. Замість цього використано whitelist tools:
- `get_chat_context`
- `get_customer_profile`
- `search_orders`
- `search_products`
- `get_inventory`
- `list_locations`
- `send_chat_message`

Такий підхід обмежує зону доступу та дозволяє контролювати, які саме дані може читати або записувати AI.

### 3. Збереження AI-повідомлень
Створено спільний сервіс `chatMessageService`, який:
- зберігає повідомлення в колекції `messages`;
- додає поле `source: "ai_admin"` для AI-відповідей;
- зберігає `meta` з інформацією про модель та ініціатора;
- емить socket events для обох сторін чату.

### 4. Сумісність з фронтендом
Було синхронізовано chat/history routes так, щоб:
- історія підтягувалась для будь-якого адміністратора, а не тільки для одного конкретного `adminId`;
- AI-відповіді були видимі як в адмінці, так і у віджеті користувача;
- публічний чат не розкривав користувачу ім'я конкретного адміністратора.

### 5. Frontend contract
Підготовлено готовий файл `docs/adminChatApi.ts`, який містить:
- типи для conversations, messages та AI status;
- готові `fetch`-методи для chat API;
- готові `fetch`-методи для AI API;
- socket event names для адмін-чату.

## Реалізовані API

### Адмін-чат
- `GET /api/admin/chat/conversations`
- `GET /api/admin/chat/:adminId/:userId`
- `PATCH /api/admin/chat/read/:senderId/:receiverId`
- `GET /api/admin/chat/support-admin`
- `GET /api/admin/chat/admin-id`

### AI admin
- `GET /api/admin/ai/status`
- `POST /api/admin/ai/suggest`
- `POST /api/admin/ai/reply`
- `POST /api/admin/ai/respond`

### Формат body для AI
```json
{
  "chatUserId": "guest_xxxxx_or_userId",
  "instructions": "optional admin hint",
  "historyLimit": 30
}
```

## Алгоритм роботи AI-адміна
1. Адміністратор відкриває діалог у frontend.
2. Frontend викликає `POST /api/admin/ai/suggest` або `POST /api/admin/ai/reply`.
3. Backend знаходить цільовий чат, дістає історію повідомлень і профіль користувача.
4. OpenAI отримує контекст та за потреби викликає tools.
5. Backend виконує tool calls та повертає результат назад у модель.
6. У режимі `reply` модель або tool `send_chat_message`, або fallback backend відправляє фінальне повідомлення в чат.
7. Повідомлення зберігається в БД та через socket доставляється у frontend.

## Результат
У результаті виконання лабораторної роботи розроблено повний backend-модуль AI-адміністратора для адмін-панелі магазину. Система забезпечує:
- безпечну інтеграцію OpenAI;
- контрольований доступ до даних з БД;
- запис AI-повідомлень у MongoDB;
- сумісність з поточним chat/widget frontend;
- готовий frontend client для швидкої інтеграції.

## Висновок
Поставлена задача виконана. Розроблений backend дозволяє інтегрувати AI-адміністратора в існуючу систему підтримки без порушення логіки ролей і без прямого доступу моделі до бази даних. Додатково підготовлено готовий frontend contract, який скорочує час інтеграції на стороні клієнта.
